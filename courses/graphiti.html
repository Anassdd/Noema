Graphiti

A knowledge graph that doesn't just store what's true — it remembers when it was true, and when you learned it. The temporal memory layer for agents. Built to evolve: facts change, the old ones are invalidated, never erased.

1The problem: knowledge that changes

Classic RAG and most knowledge graphs are append-only snapshots. You ingest "Kendra works at Globex," and it's true forever — even after Kendra quits. When a new document says "Kendra now works at Acme," a naïve store ends up with both facts and no idea which one is current. The agent contradicts itself.

Graphiti is built for the opposite case: a world where facts have a lifespan. Its whole design answers two questions a static graph can't:

"What is true now?"

The agent needs the current state — Kendra likes Nike — without being polluted by superseded facts.

"What was true then?"

But history must stay queryable — "what did Kendra like last spring?" — for reasoning, audit, and provenance.

It does this incrementally: each new piece of information is folded into the existing graph with one set of LLM calls — no re-indexing the whole corpus. That incremental-and-temporal combination is exactly what an evolving memory needs.

2The data model: episodes, entities, facts

Everything in Graphiti is one of three things, arranged in layers:

TypeWhat it isExample
EpisodicNodeA raw unit of input you fed in — a message, a document chunk, a JSON record — stamped with its real-world time. The source of truth; entities and facts point back to the episode they came from (provenance)."Kendra now prefers Nike." @ Aug 21
EntityNodeA thing the LLM extracted and resolved (deduplicated) — a person, brand, place, concept. Carries a name, a summary, and an embedding.Kendra, Nike, Acme
EntityEdgeA fact: a relationship between two entities, expressed as a triplet. This is where the temporal metadata lives. Also has an embedding (of the fact text) for search.Kendra —PREFERS→ Nike
CommunityNodeA cluster of densely-related entities with an LLM summary — a higher-level "neighborhood" view (built later, optional)."Kendra's brand affinities"

The unit you reason over is the triplet: (entity) —[fact]→ (entity). "Kendra loves Adidas shoes" becomes Kendra —LOVES→ Adidas. Simple — but the edge carries four timestamps, and that's the whole trick.

3The centerpiece: two independent clocks

A normal database has one clock — "when did this row get written." Graphiti runs two, and this bi-temporal model is the single idea that makes it special. Every fact (edge) records both:

⏱ Valid time — "when is this true in the world?"

valid_atthe fact became true in reality (Kendra started liking Nike)
invalid_atthe fact stopped being true (she moved on) — null while still current

🗄 Transaction time — "when did we learn it?"

created_atthe moment Graphiti ingested the fact
expired_atthe moment Graphiti marked it superseded by newer ingestion

Why two clocks and not one? Because "when something happened" and "when you found out" are different, and you need both. A document you ingest today might tell you about a job change from last year — valid time says "last year," transaction time says "today." Audit, conflict-resolution, and "what did we believe at the time" all need that separation.

The golden rule — invalidate, don't delete. When a new fact contradicts an old one, Graphiti does not remove the old edge. It sets invalid_at (and expired_at) on it. The edge stays in the graph, struck through, fully queryable. That's what makes history non-lossy and makes "as of date X…" questions answerable. You'll watch it happen in the next section.

5▶ Query across time — the "as of" slider

Here's the payoff of two clocks. Drag the slider to ask the graph "what did Kendra like as of this date?" Facts light up only while they're valid — the Adidas fact glows until August, then goes dark as Nike takes over. The history never disappears; you're just choosing which moment to look from.

How search() actually finds the entry points

The temporal filter above sits on top of Graphiti's hybrid retrieval — there's no single index. A query fires three searches at once and fuses them:

LensFinds facts by…
semanticcosine similarity on the embedding of the fact text — catches "footwear she's into" even with no shared words.
BM25full-text keyword match on names and fact text — exact terms, fast, precise.
graph traversalbreadth-first / node-distance from a centre node — "things near Kendra," even if textually unrelated. This is the multi-hop lens.

The three result lists are merged by a reranker — and notably without an LLM in the loop, so it's cheap and fast. You choose the strategy per query: RRF (reciprocal rank fusion, the balanced default), MMR (maximal marginal relevance, kills near-duplicates), a cross-encoder (most accurate, heavier), node-distance (rank by hops from a focus entity), or episode-mentions (recency/frequency). The temporal validity filter is then applied to whatever the rerank returns.

6Communities — the zoomed-out view

On top of entities and facts, Graphiti can build communities: clusters of densely-connected entities, each given an LLM-written summary. It uses label propagation — a cheap, incremental clustering method where each node iteratively adopts the most common label among its neighbours until labels stabilise. The point of "incremental" matters here: when a new episode arrives, Graphiti can extend existing communities dynamically instead of recomputing every cluster from scratch (contrast that with GraphRAG's heavier Leiden re-run — see the sister course). Communities give the agent a "neighbourhood-level" answer for broader questions without scanning every node.

7Backends & deployment — the part that decides if you can ship it

Graphiti is a Python library that drives a graph database underneath. As of late-2025 it supports four:

BackendShapeNotes
Neo4jserver (5.26+)The classic, most-tested driver. Needs a running database service.
FalkorDBserver (1.1.2+)Redis-based, lightweight server.
Kùzuembedded / in-process (0.11.2+)No server — links into your Python process like SQLite. The deployment-friendly option.
Amazon Neptunemanaged cloudAWS-hosted.
Why this table matters for a locked-down environment. The headline knock on Graphiti used to be "it needs Neo4j running" — a non-starter where you can't stand up a database server. The Kùzu embedded backend changes that: pip-installable, in-process, no service to provision — Azure-tenant-safe. (Heads-up: Kùzu support is newer and the embedded C-extension has had rough edges on Windows in some versions — verify the exact version on the real prod box before committing.)

LLM and embedding access is pluggable (OpenAI, Azure OpenAI, Anthropic, Gemini, Groq, local) — so the same provider-abstraction discipline you already use slots straight in.

8Where Graphiti fits in Noema

From our own SOTA research, the verdict is specific: Graphiti is the strongest candidate for the evolution layer, not the v1 multi-hop retrieval lens. Its native bi-temporal "invalidate-don't-delete" model is exactly the validated pattern for a self-cleaning, evolving memory — the Phase-2 thesis. For raw multi-hop retrieval it's "good," while HippoRAG-2 is "best."

  • Phase 1 (now): prove the extract→graph→retrieve pipeline on a simple in-process substrate; measure the graph's real lift on the French corpus.
  • Phase 2 (evolution): Graphiti becomes the leading pick — its temporal model gives you conflict-resolution and "as of" history for free, and the Kùzu embedded backend keeps it shippable in the tenant.

It plugs in behind the same search() → ScoredChunk seam, so it's a swap, not a rewrite — and every fact already carries the provenance you need to cite.
