Noema · Contextual RAG

Every chunk remembers where it came from

Cutting a document into chunks throws away context, so search matches them poorly. Contextual RAG fixes this: before embedding, an LLM writes a one-line blurb situating each chunk in its whole document. Anthropic's method, built as Noema's precise-passage memory.

✕ A raw chunk, out of context
“This approach cut the error rate by 12% over the baseline.”
Which approach? Which baseline? Embeddings & keywords can't tell — the match is weak.
✓ The same chunk, contextualized
[From §4 of the CamemBERT paper, on French NER fine-tuning:] “This approach cut the error rate by 12% over the baseline.”
Now it's findable by the terms someone would actually search.

Chunk

3

Contextualize

The LLM sees the whole document + this chunk and writes a 1–2 sentence blurb placing the chunk in the document. The blurb is prepended to the chunk.

blurb · gpt-5.4-mini
4

Embed + index

The blurb + chunk is embedded as a vector and keyword-indexed with BM25. The blurb aids findability; the original text is what gets cited.

vectors · text-embedding-3-large
5

Store

Written into the vector store (Chroma), one collection per domain. Adding documents later just appends — no rebuild.

Incremental & on-disk — no server, no GPU
Answering a question — every time, at query

Question

The user's question is embedded with the same model the chunks used.

Dense search

Nearest vectors — matches meaning, even with different wording.

BM25 search

Keyword match — catches exact terms embeddings blur (symbols, names, French words).

⚖

Fuse (RRF)

The two ranked lists merge into one — a chunk found by both rises to the top.

Optional reranker reorders the top few by true relevance

Grounded answer

The trick that makes it cheap: the whole document is placed first in every prompt, so after the first chunk it is served from the endpoint's prompt cache at ~10% price. Contextualizing a 100-chunk paper costs roughly one full read of the document, not a hundred.
